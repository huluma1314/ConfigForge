# Source Workbench Design

## Goal

把 `ConfigForge` 从“单次配置/订阅转换工具”扩展成一个可长期维护来源集合的纯前端工作台。第一阶段只聚焦：

- 维护多个来源项
- 支持启用 / 禁用 / 排序 / 删除 / 编辑
- 同时接收订阅链接、订阅文本、单节点
- 将启用来源合并后导出为一份 `Clash / Mihomo` 配置

本轮不做自动地区分组、不做复杂模板注入、不做服务端存储。

## Why This Exists

当前仓库已经能解析配置、转换订阅、展开远程资源，但工作模型仍偏“一次性转换”。这和真实使用场景不匹配：用户往往长期维护一组订阅和零散单节点，并且经常临时禁用某几个来源、替换失效订阅、保留暂时不用的节点来源。

缺少“来源管理层”会带来几个问题：

- 多个订阅无法作为独立对象长期保存和维护
- 单节点只能临时粘贴，不能和订阅并列管理
- 禁用某个来源只能靠手工删改输入内容
- 输出结果缺少“哪些来源参与了本次导出”的透明度

## Scope

第一阶段只覆盖一个新模式：`来源工作台`。

它的职责是：

- 管理来源项集合
- 解析每个来源里的节点
- 将启用来源的节点合并
- 生成一份可下载的 `Clash / Mihomo` 配置
- 导入 / 导出来源定义 JSON，避免仅依赖浏览器本地存储

它不负责：

- 自动按地区把节点塞进多个策略组
- 自动继承用户现有主配置模板
- 后端同步、云存储、跨设备共享
- 节点可用性探测
- 将现有“完整配置转换”或“订阅链接”页里的临时输入一键迁移到工作台

## Architecture

### 1. Source Model

新增独立的来源实体，而不是继续把“订阅”当成一次性输入框内容。

每个来源项至少包含：

- `id`
- `name`
- `kind`：`url` / `text` / `single`
- `enabled`
- `content` 或 `url`
- `order`
- `lastProxyCount`
- `lastParsedAt`
- `lastError`
- `lastFetchMode`：`direct` / `cors-fallback` / `none`

设计原则：

- `id` 使用 UUID 生成，避免排序、删除、导入后出现位置相关主键问题
- `单节点` 也作为来源项处理，而不是单独走特殊分支
- 启用状态只影响导出，不影响来源保留
- 解析结果只保留在运行时内存，不写入 `localStorage`
- 持久化层只保存来源定义和摘要字段，避免本地存储被大量节点撑满

运行时状态额外维护：

- `parsedProxies`
- `parseLog`
- `dirty`

其中 `dirty` 的含义固定为：来源定义已变更，但当前解析结果或导出结果尚未基于最新定义重新生成。

这样可以把“可持久化定义”和“临时解析缓存”明确分层。

### 2. Parsing Flow

来源工作台复用现有解析能力，优先复用：

- `subscription.ts`
- `parsers/uri.ts`
- `parseConfig(...)`

三类来源的解析策略：

- `url`：拉取远程订阅内容，再走现有订阅解析逻辑
- `text`：将粘贴内容视作订阅内容，支持 base64 URI、URI 列表、配置片段
- `single`：优先按单条 URI 解析，也允许 QX / Surge 的单节点行

每条来源单独记录解析成功或失败，不因为一条来源失败而中断整个工作台。

### 3. Remote Fetch and CORS

`url` 类型来源默认采用现有仓库已经在用的两级拉取策略：

1. 直接请求原始订阅 URL
2. 若失败，再退回到现有的 CORS 兜底入口

第一阶段不引入新的 service worker、浏览器扩展依赖或后端服务。

行为约束：

- 如果直接拉取成功，状态记为 `direct`
- 如果只有兜底入口成功，状态记为 `cors-fallback`
- 如果两者都失败，来源项保留但标记错误，并提示用户可以改为粘贴订阅内容

这样可以最大化复用当前 `subscription.ts` 的能力，同时把纯前端环境下 URL 来源的限制说清楚。

### 4. Refresh and Generate Triggers

第一阶段采用“显式触发”而不是“实时自动重算”：

- 编辑 `text` / `single` 来源后，保存时立即本地解析
- `url` 来源不会后台自动轮询
- 每条 `url` 来源提供手动“刷新”动作
- 工作台提供“刷新全部启用 URL 来源”动作
- 右侧合并输出通过“重新生成”按钮显式触发

这样设计的原因：

- 避免频繁网络请求
- 避免多个 URL 来源在每次输入时反复拉取
- 让用户明确知道当前输出是否基于最新订阅内容

页面加载后：

- `text` / `single` 来源采用 lazy 解析，而不是页面打开时全部 eager 重建
- 仅在以下时机重新解析：选中来源进入编辑、手动保存、或参与最终导出
- `url` 来源默认显示“需要刷新”或“使用上次摘要”的状态，不自动联网

### 5. Naming Defaults

来源名称允许留空，但保存时会自动补全默认名：

- `url`：优先使用 URL hostname
- `text`：使用 `Text Source N`
- `single`：优先使用首个成功解析节点名，否则使用 `Single Node N`

默认名始终可编辑。

### 6. Merge Model

合并只针对 `enabled === true` 的来源项。

合并过程：

1. 收集所有启用来源的 `parsedProxies`
2. 过滤空结果和解析失败来源
3. 对节点做去重
4. 生成一个仅包含节点和基础策略组的 `ConfigIR`
5. 调用现有 `generateClash(...)` 导出

第一阶段的去重策略：

- 使用复合身份键，而不是仅按名称去重
- 默认组合：`type + server + port + authIdentity + network + path + host`
- `authIdentity` 按协议取对应主认证字段：
  - `vmess` / `vless`：`uuid`
  - `ss` / `trojan`：`password`
  - `hysteria` / `hysteria2`：`password` 或 `extra.auth`
  - `http` / `https` / `socks5`：`username + password`
- 当两个来源产生相同身份键节点时，保留排序更靠前来源的版本
- 如果身份键相同但名称不同，保留优先来源的名称

这个策略优先保证“用户手动排在前面的来源更有优先级”。

### 7. Output Shape

第一阶段导出的 `Clash / Mihomo` 结构保持简单稳定：

- 一组可直接导入的顶层默认字段
- `proxies`
- 一个主策略组，例如 `PROXY`
- 一条 `MATCH,PROXY`

默认顶层字段由工作台直接生成，不要求用户自己再手工补模板。第一阶段先写死一组保守默认值，例如：

- `mixed-port`
- `allow-lan: false`
- `mode: rule`
- `log-level: info`
- `ipv6: true`

默认策略组定义：

- 名称：`PROXY`
- 类型：`select`
- 内容：所有合并后的节点名

这样做的目的不是模拟复杂主配置，而是先确保：

- 多来源可以共存
- 单节点可以混入
- 禁用来源后输出会即时收缩
- 导出结果能直接被 `Clash / Mihomo` 客户端导入

### 8. Local Persistence

来源工作台的数据保存在浏览器 `localStorage`。

原因：

- 保持纯前端、单 HTML、离线可用
- 不引入新的服务端或认证需求
- 对来源管理场景已经足够

存储内容包括：

- 来源列表
- 当前选中的来源
- 最近一次导出目标格式偏好
- 最近一次来源摘要，例如 `lastProxyCount` / `lastParsedAt` / `lastError`
- 必要时的容量告警标记

明确不存储：

- `parsedProxies`
- 原始远程订阅响应内容
- 合并后的最终输出

此外，第一阶段提供来源定义 JSON 的导入 / 导出：

- 导出只包含来源定义和摘要
- 导入时忽略运行时缓存字段
- 如果导入内容与现有来源 `id` 冲突，默认给导入项重新生成新 UUID，并作为新来源追加
- 作为 `localStorage` 意外丢失时的人工备份手段

容量约束：

- `text` 来源内容会参与持久化，因此工作台需要在保存前估算总体存储体积
- 当接近 `localStorage` 容量上限时，阻止继续保存并给出明确提示
- 提示用户优先把大体积 `text` 来源改为 `url` 来源或导出 JSON 备份后清理

### 9. State Management

来源工作台继续沿用当前项目的轻量模式，不新增状态管理库。

实现方式：

- 使用单一 TypeScript store 模块维护来源集合和当前选中项
- 通过显式渲染函数刷新 DOM
- 不引入 Zustand、Redux 或框架级响应式系统

这样能和现有项目的无框架结构保持一致，减少不必要的架构跳跃。

## UI Design

新增第三个顶级模式：`来源工作台`。

页面分成三块：

### Left: Source List

展示来源卡片列表，每条卡片显示：

- 名称
- 类型
- 启用状态
- 节点数
- 最近错误或成功状态

支持操作：

- 新增
- 编辑
- 删除
- 启用 / 禁用
- 上移 / 下移
- 刷新单条 URL 来源
- 导入来源定义 JSON
- 导出来源定义 JSON

删除动作需要二次确认，避免误删大量来源定义。

### Middle: Source Editor

用于编辑当前来源项：

- 名称
- 来源类型
- URL 或文本内容

并显示：

- 原始内容
- 解析日志
- 当前来源解析出的节点预览

### Right: Merged Output

展示启用来源合并后的结果：

- 启用来源数
- 合并后节点数
- 去重后节点数
- 导出日志
- 最终 Clash / Mihomo 预览

支持：

- 复制输出
- 下载输出
- 一键重新生成

## Responsive Strategy

第一阶段采用桌面优先设计，但不把窄屏完全放弃。

- 宽屏下使用三栏布局
- 中等宽度下切成上下堆叠，顺序为来源列表、来源编辑、合并输出
- 移动端只保证基础可用，不追求桌面级高密度编辑体验

这意味着手机上可以用，但主要优化目标仍是桌面浏览器。

## Error Handling

错误分两层：

### Source-Level Errors

单条来源可能出现：

- 拉取失败
- 直接拉取失败但 CORS 兜底成功
- 内容为空
- 解析不到节点
- URI 或配置格式非法

这些错误只标记在该来源项上，不阻塞其他来源继续参与导出。

### Export-Level Errors

以下情况阻止最终导出：

- 没有任何启用来源
- 启用来源全部解析失败
- 合并后没有可用节点
- Clash 生成器异常

这类错误显示在右侧合并输出区域。

## Testing

新增测试至少覆盖：

1. 多个来源混合导出为 Clash
2. 禁用来源不会进入输出
3. 单节点来源能和订阅来源一起合并
4. 同名或同地址节点去重逻辑符合优先级
5. 某个来源解析失败时，其它来源仍可导出

## Future Extensions

第一阶段完成后，后续可在同一模型上继续加：

- 固定策略组模板
- 自动地区识别与分组
- 将节点并入现有主配置模板
- 手动标记某来源属于某个组池

这些能力都应建立在当前“来源项集合”模型上，而不是重新回到一次性输入框模式。
