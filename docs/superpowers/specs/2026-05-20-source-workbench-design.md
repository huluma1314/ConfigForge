# Source Workbench Design

## Goal

把 `ConfigForge` 从“单次配置/订阅转换工具”扩展成一个可长期维护来源集合的纯前端工作台。第一阶段只聚焦：

- 维护多个来源项
- 支持启用 / 禁用 / 排序 / 删除 / 编辑
- 同时接收订阅链接、订阅文本、单节点
- 将启用来源合并后导出为一份 `Clash / Mihomo` 配置

本轮不做自动地区分组、不做复杂模板注入、不做服务端存储。

## Why This Exists

当前仓库已经能解析配置、转换订阅、展开远程资源，但工作模型仍偏“一次性转换”。这和真实使用场景不匹配：用户往往长期维护一组订阅和零散单节点，并且经常临时禁用某几个来源、替换失效订阅、保留暂时不用的节点来源。

缺少“来源管理层”会带来几个问题：

- 多个订阅无法作为独立对象长期保存和维护
- 单节点只能临时粘贴，不能和订阅并列管理
- 禁用某个来源只能靠手工删改输入内容
- 输出结果缺少“哪些来源参与了本次导出”的透明度

## Scope

第一阶段只覆盖一个新模式：`来源工作台`。

它的职责是：

- 管理来源项集合
- 解析每个来源里的节点
- 将启用来源的节点合并
- 生成一份可下载的 `Clash / Mihomo` 配置

它不负责：

- 自动按地区把节点塞进多个策略组
- 自动继承用户现有主配置模板
- 后端同步、云存储、跨设备共享
- 节点可用性探测

## Architecture

### 1. Source Model

新增独立的来源实体，而不是继续把“订阅”当成一次性输入框内容。

每个来源项至少包含：

- `id`
- `name`
- `kind`：`url` / `text` / `single`
- `enabled`
- `content` 或 `url`
- `order`
- `parsedProxies`
- `proxyCount`
- `lastParsedAt`
- `lastError`

设计原则：

- `单节点` 也作为来源项处理，而不是单独走特殊分支
- 启用状态只影响导出，不影响来源保留
- 解析结果缓存到来源项上，减少重复解析并提升 UI 可见性

### 2. Parsing Flow

来源工作台复用现有解析能力，优先复用：

- `subscription.ts`
- `parsers/uri.ts`
- `parseConfig(...)`

三类来源的解析策略：

- `url`：拉取远程订阅内容，再走现有订阅解析逻辑
- `text`：将粘贴内容视作订阅内容，支持 base64 URI、URI 列表、配置片段
- `single`：优先按单条 URI 解析，也允许 QX / Surge 的单节点行

每条来源单独记录解析成功或失败，不因为一条来源失败而中断整个工作台。

### 3. Merge Model

合并只针对 `enabled === true` 的来源项。

合并过程：

1. 收集所有启用来源的 `parsedProxies`
2. 过滤空结果和解析失败来源
3. 对节点做去重
4. 生成一个仅包含节点和基础策略组的 `ConfigIR`
5. 调用现有 `generateClash(...)` 导出

第一阶段的去重策略：

- 优先使用 `name`
- 若名称冲突，再回退到 `type + server + port + uuid/password`
- 当两个来源产生同 key 节点时，保留排序更靠前来源的版本

这个策略优先保证“用户手动排在前面的来源更有优先级”。

### 4. Output Shape

第一阶段导出的 `Clash / Mihomo` 结构保持简单稳定：

- `proxies`
- 一个主策略组，例如 `PROXY`
- 一条 `MATCH,PROXY`

默认策略组定义：

- 名称：`PROXY`
- 类型：`select`
- 内容：所有合并后的节点名

这样做的目的不是模拟复杂主配置，而是先确保：

- 多来源可以共存
- 单节点可以混入
- 禁用来源后输出会即时收缩
- 导出结果能直接被 `Clash / Mihomo` 客户端导入

### 5. Local Persistence

来源工作台的数据保存在浏览器 `localStorage`。

原因：

- 保持纯前端、单 HTML、离线可用
- 不引入新的服务端或认证需求
- 对来源管理场景已经足够

存储内容包括：

- 来源列表
- 当前选中的来源
- 最近一次导出目标格式偏好

## UI Design

新增第三个顶级模式：`来源工作台`。

页面分成三块：

### Left: Source List

展示来源卡片列表，每条卡片显示：

- 名称
- 类型
- 启用状态
- 节点数
- 最近错误或成功状态

支持操作：

- 新增
- 编辑
- 删除
- 启用 / 禁用
- 上移 / 下移

### Middle: Source Editor

用于编辑当前来源项：

- 名称
- 来源类型
- URL 或文本内容

并显示：

- 原始内容
- 解析日志
- 当前来源解析出的节点预览

### Right: Merged Output

展示启用来源合并后的结果：

- 启用来源数
- 合并后节点数
- 去重后节点数
- 导出日志
- 最终 Clash / Mihomo 预览

支持：

- 复制输出
- 下载输出
- 一键重新生成

## Error Handling

错误分两层：

### Source-Level Errors

单条来源可能出现：

- 拉取失败
- 内容为空
- 解析不到节点
- URI 或配置格式非法

这些错误只标记在该来源项上，不阻塞其他来源继续参与导出。

### Export-Level Errors

以下情况阻止最终导出：

- 没有任何启用来源
- 启用来源全部解析失败
- 合并后没有可用节点
- Clash 生成器异常

这类错误显示在右侧合并输出区域。

## Testing

新增测试至少覆盖：

1. 多个来源混合导出为 Clash
2. 禁用来源不会进入输出
3. 单节点来源能和订阅来源一起合并
4. 同名或同地址节点去重逻辑符合优先级
5. 某个来源解析失败时，其它来源仍可导出

## Future Extensions

第一阶段完成后，后续可在同一模型上继续加：

- 固定策略组模板
- 自动地区识别与分组
- 将节点并入现有主配置模板
- 导入 / 导出来源清单
- 手动标记某来源属于某个组池

这些能力都应建立在当前“来源项集合”模型上，而不是重新回到一次性输入框模式。
